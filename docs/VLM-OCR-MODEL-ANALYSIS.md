# VLM/OCR 모델 분석 보고서
> 영수증 OCR을 위한 Vision Language Model 비교 분석

**작성일**: 2026-01-22
**목적**: Travel Helper v2.0 영수증 자동 인식 기능을 위한 모델 선정

---

## 1. Executive Summary

### 1.1 분석 배경
Travel Helper 앱에 영수증 이미지를 촬영/업로드하면 자동으로 텍스트를 추출하여 가계부에 등록하는 AI 기능 도입을 위해 적합한 VLM(Vision Language Model) 모델을 분석합니다.

### 1.2 핵심 요구사항
- 다국어 영수증 인식 (일본어, 태국어, 베트남어 등 아시아 언어 포함)
- 구조화된 데이터 추출 (금액, 날짜, 상점명, 품목)
- 자체 호스팅 가능 (비용 효율성)
- 합리적인 추론 속도 (모바일 앱 백엔드)

### 1.3 최종 권장
| 단계 | 권장 모델 | 이유 |
|------|----------|------|
| **MVP (Phase 1)** | Google Vision API 또는 Claude Vision | 빠른 개발, 검증된 성능 |
| **Scale (Phase 2)** | Qwen2.5-VL-3B | 비용 효율, 검증된 OCR 성능 |
| **Future** | Qwen3-VL-2B | 32개 언어 OCR, Thinking 모드 |

## 2. VLM vs 전통 OCR 비교

### 2.1 기술 패러다임 변화

```
전통 OCR Pipeline:
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ 이미지   │ → │ 전처리   │ → │ 텍스트   │ → │ 후처리   │
│ 입력     │   │ (회전등) │   │ 인식     │   │ (구조화) │
└──────────┘   └──────────┘   └──────────┘   └──────────┘

VLM 기반 OCR:
┌──────────┐   ┌──────────────────────────────────────┐
│ 이미지   │ → │ VLM (End-to-End)                     │
│ 입력     │   │ → 구조화된 JSON/Markdown 직접 출력   │
└──────────┘   └──────────────────────────────────────┘
```

### 2.2 VLM 장점
- 복잡한 레이아웃 처리 (표, 다단 구성)
- 맥락 이해 (금액 vs 전화번호 구분)
- 다국어 혼합 문서 처리
- 단일 모델로 전체 파이프라인 대체

### 2.3 VLM 단점
- 추론 속도 (전통 OCR 대비 느림)
- GPU 리소스 필요
- 할루시네이션 가능성

## 4. 외부 API vs 자체 호스팅 비교

### 4.1 비용 분석 (월 기준)

| 방식 | 1,000건 | 10,000건 | 100,000건 |
|------|---------|----------|-----------|
| **Google Vision API** | $1.5 | $15 | $150 |
| **Claude Vision** | $15-30 | $150-300 | $1,500-3,000 |
| **OpenAI GPT-4V** | $20-40 | $200-400 | $2,000-4,000 |
| **자체 호스팅 (GPU)** | $50-100 | $50-100 | $50-100 |

> 자체 호스팅: Railway/Render A10G GPU 기준 (~$0.50-1.00/hour)

### 4.2 비용 손익분기점

```
월 처리량 vs 비용 그래프

비용($) ▲
        │                              ╱ Claude Vision
   300  │                           ╱
        │                        ╱
   200  │                     ╱
        │     ────────────────────── 자체 호스팅 (고정비)
   100  │  ╱
        │╱     ╱ Google Vision
     0  └──────────────────────────────────▶ 처리량(건)
           1K    5K    10K   20K   50K

손익분기점:
- Google Vision: ~7,000건/월
- Claude Vision: ~3,000건/월
```

### 4.3 기능 비교

| 기능 | Google Vision | Claude Vision | 자체 호스팅 |
|------|---------------|---------------|-------------|
| **정확도** | 높음 (OCR 특화) | 매우 높음 | 높음 |
| **구조화 추출** | 별도 작업 필요 | 직접 JSON 출력 | 직접 JSON 출력 |
| **다국어** | 100+ 언어 | 좋음 | 19-32개 언어 |
| **커스터마이징** | ❌ | ❌ | ✅ fine-tuning |
| **오프라인** | ❌ | ❌ | ✅ |
| **데이터 프라이버시** | ⚠️ | ⚠️ | ✅ |
| **레이턴시** | ~1-2초 | ~2-5초 | ~1-3초 |

## 6. 영수증 OCR 프롬프트 예시

### 6.1 기본 프롬프트

```
다음 영수증 이미지에서 정보를 추출하여 JSON 형식으로 반환해주세요.

필요한 필드:
- store_name: 상점명
- date: 날짜 (YYYY-MM-DD)
- time: 시간 (HH:MM, 있는 경우)
- currency: 통화 코드 (JPY, USD, EUR 등)
- total_amount: 총 금액 (숫자만)
- items: 품목 리스트 (선택)
- category_hint: 추정 카테고리 (food, transport, shopping, lodging, activity, etc)

JSON 형식으로만 응답하세요.
```

### 6.2 예상 출력

```json
{
  "store_name": "ローソン 渋谷店",
  "date": "2025-01-20",
  "time": "14:32",
  "currency": "JPY",
  "total_amount": 1250,
  "items": [
    {"name": "おにぎり", "price": 150},
    {"name": "お茶", "price": 100},
    {"name": "サンドイッチ", "price": 350}
  ],
  "category_hint": "food"
}
```

## 7. 권장 구현 로드맵

### Phase 1: MVP (1-2주)

```
목표: 빠른 검증
모델: Google Vision API 또는 Claude Vision API
이유: 인프라 구축 없이 즉시 사용 가능

구현:
1. 이미지 업로드 API
2. 외부 API 호출 → 텍스트 추출
3. LLM으로 구조화 (선택)
4. DB 저장
```

### Phase 2: 자체 호스팅 전환 (2-4주)

```
조건: 월 5,000건 이상 처리 예상 시
모델: Qwen2.5-VL-3B
이유: 검증된 성능, 비용 효율

구현:
1. GPU 서버 프로비저닝 (Railway/Render)
2. 모델 서빙 (vLLM 또는 TGI)
3. 비동기 처리 큐 (Redis + Bull)
4. 모니터링 설정
```

### Phase 3: 최적화 (4-8주)

```
목표: 성능 및 비용 최적화
옵션:
- Qwen3-VL-2B 전환 (다국어 강화)
- Fine-tuning (영수증 특화)
- INT8 양자화 (비용 절감)
- Batch 처리 최적화
```

```
현재 시점 권장: Qwen2.5-VL-3B
```
