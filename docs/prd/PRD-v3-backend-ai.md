## 1. Overview

### 1.1 배경

Travel Helper v2.0은 오프라인 전용 앱으로 로컬 SQLite에 데이터를 저장한다.
v3.0에서는 **백엔드 서버**를 도입하여 다음 기능을 추가한다:

1. **사용자 인증**: 회원가입/로그인, 소셜 로그인
2. **데이터 동기화**: 여러 기기에서 동일한 데이터 접근
3. **AI 영수증 OCR**: 영수증 사진으로 자동 지출 등록
4. **AI 챗봇**: 여행 관련 질문 응답, 지출 분석

### 1.2 목표

| 목표 | 지표 |
|------|------|
| 영수증 등록 시간 단축 | 수동 입력 대비 80% 시간 절감 |
| 사용자 편의성 향상 | 챗봇으로 지출 분석 즉시 확인 |
| 데이터 안전성 | 클라우드 백업으로 데이터 손실 방지 |

### 1.3 범위

**In Scope**:
- NestJS 백엔드 API
- JWT 기반 인증 (이메일, Apple, Google)
- AI 영수증 OCR (VLM)
- AI 챗봇 (LLM)
- PostgreSQL 데이터베이스
- 오프라인 → 온라인 데이터 마이그레이션

**Out of Scope** (v4.0 이후):
- 여행 공유/협업
- 실시간 동기화 (WebSocket)
- 프리미엄 구독 모델

## 3. Functional Requirements

### 3.1 인증 (Auth Module)

#### FR-AUTH-001: 이메일 회원가입
- 이메일, 비밀번호, 이름 입력
- 이메일 형식 검증
- 비밀번호 최소 8자, 영문+숫자 조합
- 중복 이메일 방지

#### FR-AUTH-002: 이메일 로그인
- 이메일 + 비밀번호로 로그인
- JWT Access Token (15분) + Refresh Token (7일) 발급
- 로그인 실패 시 에러 메시지

#### FR-AUTH-003: Apple 로그인
- Sign in with Apple 연동
- 첫 로그인 시 자동 회원가입
- 이름은 Apple에서 제공하는 값 사용

#### FR-AUTH-004: Google 로그인
- Google Sign-In 연동
- 첫 로그인 시 자동 회원가입

#### FR-AUTH-005: 토큰 갱신
- Refresh Token으로 새 Access Token 발급
- Refresh Token 만료 시 재로그인 요구

#### FR-AUTH-006: 로그아웃
- 클라이언트에서 토큰 삭제
- 서버에서 Refresh Token 무효화

### 3.2 AI 영수증 OCR (AI Module)

#### FR-AI-001: 영수증 이미지 분석
- 지원 포맷: JPEG, PNG, HEIC
- 최대 크기: 10MB
- 분석 결과:
  - 상점명
  - 날짜/시간
  - 품목 리스트 (이름, 가격, 수량)
  - 합계 금액
  - 통화
  - 추천 카테고리
  - 신뢰도 점수 (0-1)

```typescript
interface ReceiptAnalysis {
  store: string;
  date: string;           // YYYY-MM-DD
  time?: string;          // HH:mm
  items: {
    name: string;
    price: number;
    quantity: number;
  }[];
  subtotal?: number;
  tax?: number;
  total: number;
  currency: string;
  category: Category;
  confidence: number;     // 0.0 ~ 1.0
  rawText?: string;       // 원본 텍스트
}
```

#### FR-AI-002: 분석 결과 확인/수정
- OCR 결과를 사용자에게 표시
- 수정 가능한 필드: 금액, 카테고리, 날짜, 메모
- 확인 후 지출로 등록

#### FR-AI-003: 자동 지출 생성
- 분석 결과를 기반으로 Expense 자동 생성
- 현재 활성 여행에 연결
- 방문지는 통화 기준으로 자동 매칭

#### FR-AI-004: 배치 처리
- 여러 영수증 이미지를 한번에 업로드
- 순차적으로 분석 후 결과 목록 표시
- 일괄 확인/등록

#### FR-AI-005: 이미지 저장
- 분석된 영수증 이미지를 클라우드에 저장
- 지출 상세에서 영수증 이미지 확인 가능

### 3.3 AI 챗봇 (AI Module)

#### FR-AI-010: 지출 관련 질문
- 현재 여행의 지출 데이터를 컨텍스트로 제공
- 지원 질문 유형:
  - 지출 현황: "오늘/이번주/총 얼마 썼어?"
  - 예산 분석: "예산 얼마 남았어?", "며칠 버틸 수 있어?"
  - 카테고리 분석: "뭐에 가장 많이 썼어?"
  - 비교: "어제보다 얼마나 더 썼어?"

#### FR-AI-011: 여행 정보 질문
- LLM의 일반 지식 활용
- 지원 질문 유형:
  - 팁 문화: "일본에서 팁 줘야 해?"
  - 환율: "지금 엔화 환율 어때?"
  - 추천: "파리에서 저렴하게 먹을 수 있는 곳?"

#### FR-AI-012: 대화 히스토리
- 사용자별 대화 기록 저장
- 여행별로 분리된 대화
- 최근 20개 메시지 컨텍스트 유지

#### FR-AI-013: 스트리밍 응답
- 챗봇 응답을 스트리밍으로 표시
- 타이핑 효과로 UX 향상

#### FR-AI-014: 추천 질문
- 대화 후 관련 추천 질문 제시
- 예: "오늘 얼마 썼어?" → "이번 주 총 지출은?" 추천

### 3.4 데이터 동기화 (Sync Module)

#### FR-SYNC-001: 초기 마이그레이션
- 로그인 후 로컬 데이터를 서버에 업로드
- 충돌 해결: 서버 데이터 우선 (사용자 선택 가능)
- 마이그레이션 진행률 표시

#### FR-SYNC-002: 오프라인 모드
- 인터넷 없이도 앱 사용 가능
- 로컬 SQLite에 임시 저장
- 온라인 복귀 시 자동 동기화

#### FR-SYNC-003: 변경 사항 동기화
- 앱 시작 시 서버와 동기화
- Pull-to-refresh로 수동 동기화
- 백그라운드 동기화 (선택)

## 5. Non-Functional Requirements

### 5.1 성능

| 항목 | 요구사항 |
|------|----------|
| API 응답 시간 | p95 < 200ms (일반 API) |
| OCR 처리 시간 | < 5초 (단일 영수증) |
| 챗봇 응답 시간 | First token < 1초 |
| 동시 사용자 | 1,000 CCU 지원 |

### 5.2 가용성

| 항목 | 요구사항 |
|------|----------|
| Uptime | 99.9% (월간) |
| 오프라인 모드 | 인터넷 없이도 핵심 기능 동작 |
| 장애 복구 | RTO < 1시간 |

### 5.3 보안

| 항목 | 요구사항 |
|------|----------|

| 데이터 암호화 | TLS 1.3 (전송), AES-256 (저장) |
| 비밀번호 | bcrypt (salt rounds: 12) |
| API Rate Limiting | 100 req/min per user |

### 5.4 확장성

| 항목 | 요구사항 |
|------|----------|
| AI Provider | 추상화로 교체 가능 |

## 6. AI Provider Strategy

### 6.1 Phase 1: 자체 호스팅 (MVP)

| 기능 | 모델 | 호스팅 |
|------|------|--------|
| 영수증 OCR | Qwen2-VL-7B | 자체 서버 (FastAPI) |
| 챗봇 | Qwen2.5-7B-Instruct | 자체 서버 (FastAPI) |

**장점**: 비용 절감, 학습 목적
**단점**: 인프라 관리 필요, GPU 서버 비용

### 6.2 Phase 2: 외부 API (성장기)

| 기능 | Provider | 모델 |
|------|----------|------|
| 영수증 OCR | Groq | Llama 3.2 Vision |
| 챗봇 | Groq | Llama 3.3 70B |

**장점**: 안정성, 스케일, 관리 부담 감소
**단점**: API 비용 발생

### 6.3 Phase 3: 다시 자체 호스팅 (확장기)

- 사용자 증가 시 API 비용 > 자체 호스팅 비용
- GPU 서버 확장 or 클라우드 GPU 활용
- 모델 최적화 (quantization, distillation)

### 6.4 Provider 추상화

  chat(messages: ChatMessage[], context?: UserContext): Promise<string>;
  chatStream?(messages: ChatMessage[], context?: UserContext): AsyncIterable<string>;
}

## 7. Data Migration

### 7.1 오프라인 → 온라인 마이그레이션

```
┌──────────────────────────────────────────────────────────┐
│                    마이그레이션 Flow                       │
└──────────────────────────────────────────────────────────┘

1. 로그인/회원가입 완료
         │
         ▼
2. 로컬 데이터 존재 확인
         │
    ┌────┴────┐
    │ 있음    │ 없음
    ▼         ▼
3. 마이그레이션   4. 정상 시작
   프롬프트
         │
         ▼
5. "기존 데이터를 클라우드에
    업로드하시겠습니까?"
    [업로드] [나중에]
         │
         ▼
6. 업로드 진행
   - 여행 데이터
   - 지출 데이터
   - 지갑 데이터
         │
         ▼
7. 충돌 발생 시
   "서버에 동일한 여행이 있습니다."
   [서버 데이터 유지] [로컬 데이터로 덮어쓰기]
         │
         ▼
8. 완료
   "마이그레이션이 완료되었습니다!"
```

### 7.2 충돌 해결 정책

| 시나리오 | 기본 동작 |
|----------|----------|
| 같은 ID의 여행 존재 | 서버 데이터 유지 (사용자 선택 가능) |
| 같은 ID의 지출 존재 | 최신 updatedAt 기준 |
| 로컬에만 있는 데이터 | 서버에 업로드 |
| 서버에만 있는 데이터 | 로컬에 다운로드 |
